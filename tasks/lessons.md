# Lessons Learned

Self-improvement log. After any correction from the user, record the pattern and a rule to prevent the same mistake. Review at session start.

## Rules

1. **Don't sell the work short.** When summarizing project status, include the intellectual journey — the wrong hypotheses, course corrections, and key insights — not just the final metrics. A 4-phase project that discovered "it's the labels, not the loss" through 87 experiments is fundamentally different from "99% accuracy, done." *Source: F2 status recap, user correction.*

2. **Verify claims against actual code/docs before stating them.** Stated Phase 3 trained 3 models (it trained 1). Stated BT baseline was not considered effective (it was the winner). Both were wrong and contradicted what we'd just read. When uncertain, re-read the source rather than relying on memory. *Source: F2 recap discussion, user caught both errors.*

3. **Don't contradict your own earlier findings.** Called the diversity knob "a simple slider that gets you most of the way there" after previously establishing that its hardcoded coefficients (favorite + 0.8*repost + 0.5*follow) have no empirical basis. Recognize when a new framing contradicts an earlier one and reconcile them. *Source: Pareto frontier discussion, user flagged the contradiction.*

4. **Read source docs thoroughly, not just agent summaries.** Agent summaries can miss or mischaracterize details. When accuracy matters (project status, phase history), read the actual files. *Source: multiple corrections in F2 recap.*

5. **NumpyEncoder must handle ALL numpy types from day one.** Every `NumpyEncoder` class must include `np.bool_`, `np.floating`, `np.integer`, and `np.ndarray`. This has caused serialization failures multiple times. When writing any JSON encoder for numpy data, always include the full set — don't wait for it to break. *Source: Exp 4 partial sampling, repeated serialization failures across sessions. Happened again in RecSys Phase 1.*

6. **Success criteria need both tests AND scripts.** Tests (pytest) verify correctness with pass/fail assertions. Scripts produce inspectable artifacts (JSON, printed tables) that humans review. Phase verification needs both: tests catch regressions, scripts produce the numbers you'll reference in the paper. Don't put all verification in tests/ alone. *Source: RecSys Phase 1, user correction.*

7. **Follow the full lifecycle even under deadline pressure.** PLAN → TEST → IMPLEMENT → RETRO. Skipping retro loses the learnings (e.g., the user-platform near-alignment surprise in RecSys Phase 1 directly affects Phase 2 framing). Skipping verification scripts means no inspectable output. The discipline exists because each step catches things the others miss. *Source: RecSys Phase 1, user correction.*

8. **Use importlib for modules behind Phoenix/grok import chains.** `enhancements/data/__init__.py` and `enhancements/reward_modeling/__init__.py` eagerly import modules that depend on vendored Phoenix/grok. Any new code or tests that only needs a subset (e.g., `movielens.py`, `alternative_losses.py`) must use `importlib.util.spec_from_file_location()` to load modules directly, bypassing `__init__.py`. *Source: RecSys Phase 1, import chain failures in both module and tests.*

9. **One phase = one branch. Commit, push, wait for merge.** Each phase (a reasonable chunk of TDD work to completion) gets its own branch off main. At the end of the phase: commit, push, stop. The user creates the PR and merges to main. Only create the next branch off main after the user gives the green light. Do NOT start the next phase on the same branch, do NOT pile multiple phases into one branch, do NOT work without committing. *Source: RecSys Phases 1-3, where 3 phases accumulated uncommitted on the wrong branch (`paper/f2-preprint`). Cost: messy stash/checkout, conflict resolution on files that don't belong on the branch.*

10. **Branch at the moment scope changes, not when you notice.** When the user says "let's take a detour" / "let's expand scope" / "new phase" — that is the trigger to branch. Not after you've already written files on the wrong branch. The right sequence is: hear scope change → stop → commit or stash current work → branch → then write code. Waiting until you "notice" the discipline lapse means you've already written code, the commit becomes a tangled cleanup, and the user has to call you out. *Source: Tier 2 dataset expansion detour, where Phase A download scripts were written on `recsys/paper-writing` instead of a new branch. Lesson 9 covers the rule; this lesson covers the trigger.*

11. **When a phase ends, the very next action is the retro — not the next phase, not "what's next?"** Lesson 7 already says "follow the full lifecycle." This one is tighter: the moment the last phase task completes, the next thing you do is create `docs/fN/{phase}_retro.md` with the 6-section format. Not commit and ask "what's next", not summarize results in chat, not move to the next phase. Retro first, everything else after. Embedding retro-flavored content in commit messages or plan-doc updates is NOT a retro — it doesn't produce a reviewable artifact and doesn't force the 6-section structure that surfaces Surprises and Implicit Assumptions. *Source: Tier 2 Phase A. Lesson 7 covers "don't skip retro"; this one answers the failure mode of "I thought the commit-message/plan-doc update counted." It doesn't. Retros need their own file.*

12. **Adapter methods that hide a default argument are foot-guns. Make implicit defaults visible in the name.** A `LoadedDataset.generate_preferences(weights, n, seed)` method that quietly uses `self.pool` is convenient until a caller needs to use a different pool — then the silent fallback corrupts results without raising an error. Group C of `labels_not_loss` trains on a temporal subset (`early_pool`), passed `early_pool` as a `features` parameter into `_train`, and `_train` then called `ds.generate_preferences()` which discarded `features` and used `ds.pool`. The byte-equivalence regression test caught it within 2 minutes; without that test it would have shipped silently. Either remove such adapters and force callers to pass everything explicitly, OR rename them to make the implicit argument visible (`generate_preferences_on_default_pool()`). **Why:** an "obvious" convenience adapter can hide bugs that are extremely hard to detect downstream. **How to apply:** when designing a wrapper that takes a subset of an underlying function's arguments, ask "what happens if a caller wants to override the implicit arg?" If the answer is "they have to bypass the wrapper," the wrapper's name must telegraph that. *Source: Phase B Commit 3, labels_not_loss Group C drift.*

13. **Byte-equivalence on a fixed-K test dataset is NECESSARY but NOT SUFFICIENT for a multi-K refactor.** Phase B's regression test on ml-100k (K=3 stakeholders) proved 7/7 scripts byte-equivalent and let me declare the refactor complete. Phase C's first run on mind-small (K=5) immediately surfaced **three** Phase B misses, all K-dependent: (1) a hardcoded output filename that worked fine on ml-100k because ml-100k was the only caller, (2) a function literally named `compute_hypervolume_3d` with `rng.uniform(..., size=(n_samples, 3))`, and (3) a dataset-key heuristic `list(data.keys())[0]` that happened to short-circuit correctly on ml-100k but picked the wrong key on any non-MovieLens dataset. None of these could have been caught by byte-equivalence on ml-100k because the buggy code paths only activate when K ≠ 3 or when dataset_name != "movielens". **Why:** byte-equivalence tests the invariance of a SPECIFIC execution path, not the GENERALITY of the refactor. **How to apply:** when refactoring a script from "hardcoded to one value of K" to "supports any K," the regression test must run on at least TWO values of K before you can claim the refactor is done. If you only have access to K=3 data, at minimum grep the refactored code for every literal `3` and every hardcode of a stakeholder name; that catches most K-dependent bugs statically. *Source: Phase C launch, 3 Phase B misses surfaced within the first hour of running on mind-small (K=5). Lesson 12 covers adapter design; this lesson covers refactor verification methodology.*

14. **Output-JSON string literals are a separate class of K-dependent bug. Grep them too.** Phase B shipped a 4th K-dependent miss that neither the Phase C launch nor Phase C's additional static checks caught. `run_extremal_evidence.py` line 325 hardcoded `"target": "user"` inside the output JSON's `selection_concentration.config` block — not in any computation, just a metadata field written to disk. This bug is invisible under byte-equivalence on ml-100k because ml-100k's first stakeholder IS "user", so the literal value is correct by coincidence. It's also invisible under most static grep patterns because "user" is a common English word and grep noise dominates. The only way to catch it pre-flight is to grep for **output JSON literals** specifically — anywhere in the refactored script that assigns a stakeholder-name-shaped string to a dict key that will be serialized. Three of the four Phase B K-dependent misses lived in output literals, not in computational logic: (1) filename literal, (3) dataset-key literal in a reader, (4) target metadata literal. Pattern: **output-path strings and metadata-field defaults are the most common K-dependent hiding spots after a multi→single dataset refactor**. **Why:** output literals aren't exercised numerically, so byte-equivalence on a single dataset short-circuits them whenever the literal value happens to equal the correct one for that dataset. **How to apply:** after any multi→single dataset refactor, do a targeted grep of the script for `".*json"` (filename literals), `"[a-z_]+": "[a-z_]+"` (dict-literal metadata), and `"[a-z]+\_utility"` (per-stakeholder-named field names). Anything that looks like a stakeholder name appearing as a string literal needs to be either parameterized on the dataset spec or replaced with `ds.spec.*` lookups. This grep pattern would have caught Phase B miss #4 in 30 seconds. *Source: Phase D planning, caught the 4th Phase B miss pre-flight by critical-reading the script's `main()` output assembly. Lesson 13 covers "needs more than one K"; this lesson covers "also grep the string literals in the output side, not just the computational side."*
